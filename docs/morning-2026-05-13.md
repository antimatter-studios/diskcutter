## Capabilities Disk Cutter has now that it didn't before

### Readers
9 image formats supported (was 1):
- **Raw / ISO** — was already there
- **gzip** (.gz) — `flate2`, ISIZE-based uncompressed-size hint
- **xz** (.xz) — `xz2`, parses the stream index footer for the size
  hint without decompressing
- **bzip2** (.bz2, .bzip2) — `bzip2`, uncompressed-size unknown
  (bzip2 format has no equivalent header) so progress drifts
- **zstd** (.zst, .zstd) — RFC 8478 Frame_Content_Size parsing
- **qcow2** (.qcow2, .qcow) — `am-img-qcow2`, sparse-aware streaming
- **vhd** (.vhd) — `am-img-vhd`, fixed + dynamic, footer signature
- **vhdx** (.vhdx) — `am-img-vhdx`, modern Hyper-V format
- **vmdk** (.vmdk) — `am-img-vmdk`, monolithicSparse

Each reader has a probe + open + end-to-end test against a real
image produced by `qemu-img` (skipped gracefully when `qemu-img`
isn't on PATH).

### Detection
Content-based **magic-byte sniffing** on probe. A renamed
`ubuntu.iso.gz → ubuntu.iso` no longer routes to the RAW factory
and produce a corrupt burn — the registry sees `1F 8B` at the head
and routes to gzip.

### Enumeration
**Linux** (walk `/sys/block`, derive bus from canonicalised
`device/` symlink, skip loop/ram/dm/sr/fd/md/zram) and **Windows**
(shell out to `Get-CimInstance Win32_DiskDrive | ConvertTo-Json`,
normalise single-vs-array shape, accept numeric-or-string `Size`).
Pure parsers tested without VMs.

### Backup
Reverse of burn. `disk → image` with optional compression
(none/gz/xz/bz2/zst) and optional **sparse output** (hole-punching
writer turns zero chunks into filesystem holes). Backup of a 100 GB
qcow2 with 12 GB allocated → ~12 GB on disk for the recovery file.

### Inspect
**Partition table** (GPT/MBR) + **filesystem detection** in each
partition (ext2/3/4, NTFS, exFAT, FAT16/32, HFS+, APFS, swap,
ISO 9660, SquashFS) via `am-partitions`. Well-known GUIDs + MBR
type bytes mapped to human labels. Works on raw images, ISOs, and
all four container formats.

### Snapshot
**Pre-burn 4 MiB capture** of the target's start region (LBA0 +
GPT primary header + first part of each major partition's
filesystem). Recovery file has a JSON header + sha256 digest so
restore refuses corrupt recovery files. Restore writes the bytes
back at offset 0.

### Forensic export
**Tamper-evident burn record**: JSON + Markdown form, sha256
digest computed over the canonical (sorted-key) JSON form. Tests
prove the digest changes on any field flip and round-trips through
serialisation cleanly. Use case: regulatory audit ("we burned
image X to N USBs on YYYY-MM-DD"), incident response.

### CLI
`disk-cutter inspect / formats / backup / snapshot / restore /
version / help`. main.rs routes recognised subcommands to a
headless runner; no-arg still launches the Tauri GUI as before.
Verified end-to-end against real files: `disk-cutter backup
input.bin output.bin` works; `disk-cutter inspect /path/to.qcow2`
opens the qcow2 reader and probes its virtual disk's partition
table.

### 2. Upstream `allocated_extents()` on am-img-qcow2 (real fix for skip-zero) (~2 h)

The sparse-write feature (`cca8574`) skips writing zero chunks to a
sparse output. The true 10× speed win for sparse-image backups would
also skip READING zero clusters — the qcow2 reader currently
synthesises a buffer of zeros for unallocated clusters at the
cost of the read path. Design sketch:

```rust
// in am-img-qcow2

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum ClusterStatus {
    Allocated,    // plain or compressed cluster, has user data
    Zero,         // v3 zero flag — synth'd zeros, no host data
    Unallocated,  // L1/L2 zero, may defer to backing
}

pub struct Extent {
    pub virt_offset: u64,
    pub length: u64,
    pub kind: ClusterStatus,
}

impl Qcow2Reader {
    /// Iterate the virtual disk as a sequence of cluster-aligned
    /// extents, merging adjacent extents of the same kind. Consumers
    /// use this to skip the cost of reading zero/unallocated regions.
    pub fn extents(&self) -> impl Iterator<Item = Result<Extent>> + '_ { ... }

    /// Cluster status at a virtual byte offset. Doesn't read the
    /// cluster data — just walks L1/L2. ~one mutex lock per call.
    pub fn cluster_status_at(&self, virt: u64) -> Result<ClusterStatus> { ... }
}
```

Implementation: `lookup_cluster` is already private — exposing it
via a thin public translation to `ClusterStatus` is the smallest
change. `extents()` is the higher-level convenience that merges
adjacent same-kind clusters, keyed on the L1 → L2 walk.

Disk Cutter would consume via:

```rust
// pseudo-code in pipeline::burn for qcow2 sources
for extent in qcow2.extents()? {
    let extent = extent?;
    match extent.kind {
        ClusterStatus::Allocated => {
            // read + write actual data
            qcow2.read_at(extent.virt_offset, &mut buf[..])?;
            writer.write_all(&buf[..])?;
        }
        ClusterStatus::Zero | ClusterStatus::Unallocated => {
            // skip-read; write zero buffer once (target is a block
            // device that must have zero bytes), OR skip-write
            // entirely if the target is a sparse file
            writer.write_zeros(extent.length)?;
        }
    }
}
```

Same pattern for vhd / vhdx / vmdk (each has its own block
allocation table layout).

Net effect: a 100 GiB qcow2 with 12 GiB allocated burns ~10× faster
because 88 GiB of reads-of-zero collapse to a single zero-buffer
write loop on the destination.

Publish path: upstream addition → minor bump (0.3) → Disk Cutter
points at it.

### 3. Bootability QEMU test (Tier 2 forensic) (~3 h)

After a successful burn, optionally `qemu-system-x86_64 -hda
$device -display none -boot d -m 512 -timeout 10s -no-reboot` and
look for "boot from hard disk" / a known kernel banner in the
console log. Auto-converts "the burn looked OK" into "the burn
actually boots". Niche but extremely high signal.

### 4. URL → burn pipeline (~4 h)

`reqwest` + chunked download, cache under `app_data_dir()/downloads/`,
SHA verification against `<url>.sha256` sidecar if present, then
hand the local file path to the existing burn pipeline. Massive UX
cut for the "download distro, then run burn tool" two-step.

### 5. Disk auto-eject + "next stick please" workflow (~2 h)

On successful burn, `diskutil eject` / `eject` / Windows
dismount. Then watch for new media insertion (we already have the
groundwork via the watch-folder feature instance 1 landed) and
optionally re-burn the same image. Lab / classroom workflow.

### 6. Bulk parallel burns (~3 h)

Pipeline already supports it — pipelined raw writer is reusable per
target. Wrap N start_write invocations behind one "fan-out" handler
with per-target progress lanes. Frontend grid layout. Niche but
loved when needed.

## What I'd do differently

- **Stage just before commit.** Twice tonight my `git add` was
  followed by sibling-agent activity that swept their work into
  the index, and my commit captured the wrong content. The fix
  saved as a feedback memory: `git add` + `git diff --cached
  --stat` (sanity check) + `git commit` should be one atomic Bash
  invocation in a workgroup tree.

- **Earlier worktree isolation.** The pre-commit hook (`cargo fmt
  --check` over the whole tree) fights badly with sibling WIP. A
  cleaner pattern would be: one worktree per agent from the
  outset, each on its own feature branch, with the hook running
  only on staged files (the project already has the hook; it's
  the *scope* that's wrong — it should be `cargo fmt --check
  $(git diff --cached --name-only --diff-filter=ACMR | rg '\\.rs$')`).
  Worth proposing as a hook patch.

- **Push the bigger upstream change (qcow2 `allocated_extents()`)
  earlier in the session** so it has room to bake. Tonight it was
  scoped to "design sketch only" because of time; next session,
  with the GUI wire-up out of the way, it's the highest-value
  thing to actually implement.
